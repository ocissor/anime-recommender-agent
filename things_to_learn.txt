---

### âœ… `logging.error(msg)`

* Logs an error **message only**.
* Doesnâ€™t include the **stack trace** unless you pass `exc_info=True`.

| Only in `except` block | âŒ                          | âœ…                             |

### âœ… Best Practice:

To log **uncaught exceptions** (i.e., exceptions that are not inside a `try/except` block), you can hook into Pythonâ€™s global exception handler using `sys.excepthook`.

---

### âœ… Step-by-step Guide to Log Uncaught Exceptions

```python
import logging
import sys

# Set up logging
logging.basicConfig(
    level=logging.ERROR,
    format='%(asctime)s [%(levelname)s] %(message)s',
    filename='errors.log',  # Log to a file
)

# Define a handler for uncaught exceptions
def handle_uncaught_exception(exc_type, exc_value, exc_traceback):
    if issubclass(exc_type, KeyboardInterrupt):
        # Let Ctrl+C still interrupt
        sys.__excepthook__(exc_type, exc_value, exc_traceback)
        return
    logging.critical("Uncaught exception", exc_info=(exc_type, exc_value, exc_traceback))

# Register it
sys.excepthook = handle_uncaught_exception

# --- Test with an uncaught error ---
def do_something():
    x = 1 / 0  # This will NOT be caught in try/except

do_something()
```

---

### ğŸ§  Whatâ€™s Happening:

* `sys.excepthook` is Pythonâ€™s fallback error handler.
* By assigning a custom function, you're saying:

  > "Whenever something crashes and nobody catches it, log it for me."

---

### ğŸ“ Notes:

* This works only for **main thread**. For threads or subprocesses, you need special handling (`threading.excepthook` or `concurrent.futures`'s error callbacks).
* You can log to a file, rotate logs, or even send alerts (e.g., via email or Slack) in `handle_uncaught_exception`.

Great question! `top_p` and `top_k` are **sampling parameters** used during **text generation in LLMs** (like GPT, LLaMA, T5, etc.). They control how the model chooses the **next token** â€” especially when not using greedy decoding.

---

## ğŸ§  Why Sampling Exists

When generating text, the LLM outputs a **probability distribution** over all possible next tokens. Instead of always picking the highest probability token (greedy), we **sample** to add variety and creativity.

---

## ğŸ” `top_k`: **Token cutoff by rank**

* **Limits sampling to the top `k` most likely tokens**
* Only the `k` tokens with the highest probabilities are considered
* The rest are **ignored**

### Example:

```text
top_k = 5 â†’ only sample from the top 5 most likely next tokens
```

âœ… Leads to more diversity
âš ï¸ But can ignore good low-ranked tokens if k is too small

---

## ğŸ” `top_p`: **Nucleus Sampling**

* Also called **"nucleus sampling"**
* Instead of a fixed number of tokens, it looks at the **smallest group of tokens whose cumulative probability â‰¥ `p`**
* The model samples from that dynamic set

### Example:

```text
top_p = 0.9 â†’ sample from the smallest set of tokens that together have 90% of the total probability mass
```

âœ… Adaptively picks a variable number of tokens based on confidence
âœ… Keeps more diverse options when distribution is flat
âš ï¸ Too low = very deterministic, too high = may include poor tokens

---

## ğŸ§  TL;DR Table

| Parameter     | Meaning                           | Common Value | Controls...               |
| ------------- | --------------------------------- | ------------ | ------------------------- |
| `top_k`       | Keep only top-k tokens            | 40, 50       | Hard cutoff               |
| `top_p`       | Keep tokens until cum. prob â‰¥ p   | 0.9, 0.95    | Soft, adaptive cutoff     |
| `temperature` | Controls randomness (scale probs) | 0.7â€“1.0      | Exploration vs. precision |

---

## âœ… Suggested Defaults

For high-quality, creative generation:

```python

top_p=0.9
temperature=0.8
```

Want more creative? Increase `temperature` or `top_p`.
Want more focused? Decrease them.
